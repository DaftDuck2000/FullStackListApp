# Frontend Overview

The frontend is a single-page application (SPA) built with **React 19** and **TypeScript**, bootstrapped with **Vite 8**.

## Entry Point

File: `src/main.tsx`

```tsx
import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import App from './App.tsx'

createRoot(document.getElementById('root')!).render(
  <StrictMode><App /></StrictMode>
)
```

## Component Tree

There is **one component**: `App` in `src/App.tsx`.

```
<App>
  ├── <h1>Items</h1>
  ├── <form className="item-form">      (Add / Edit form)
  │   ├── <input> Name
  │   ├── <input> Priority
  │   ├── <input> Method
  │   ├── <button> Add
  │   └── <button> Reset
  ├── <div className="filter-bar">       (Search + Filter controls)
  │   ├── <input> Search name
  │   ├── <select> Priority filter
  │   └── <select> Method filter
  └── <ul className="item-list">
      └── <li className="item-card">     (repeated per item)
          ├── <div> item info
          └── <div> Edit / Remove buttons
```

## State Management

All state is managed with `useState` hooks in `App.tsx`:

| State Variable | Type | Purpose |
|---|---|---|
| `items` | `Item[]` | The full list of items from the API |
| `name` | `string` | Form input for item name |
| `priority` | `string` | Form input for item priority |
| `method` | `string` | Form input for item method |
| `filterName` | `string` | Name search filter input |
| `filterPriority` | `string` | Priority dropdown filter |
| `filterMethod` | `string` | Method dropdown filter |

## Interaction Flow

### Page Load
1. `useEffect(() => { loadItems() }, [])` fires on mount
2. `loadItems()` calls `GET /api/items`
3. Response is stored in `items` state
4. Items render as `.item-card` list items

### Adding an Item
1. User fills in form fields (name, priority, method)
2. Submit calls `addItem()` which calls `POST /api/items` (id is auto-generated by the backend)
3. Form fields are cleared
4. `loadItems()` is called to refresh the list

### Editing an Item
1. User clicks "Edit" on an item card
2. `editItem(id)` calls `await removeItem(id)`, then copies the found item data into the form fields
3. User modifies fields and clicks "Add" to re-submit
4. Item is re-created with potentially new data

### Deleting an Item
1. User clicks "Remove"
2. `removeItem(id)` calls `DELETE /api/items { id }`
3. `loadItems()` refreshes the list

### Resetting
1. User clicks "Reset"
2. `resetList()` calls `POST /api/reset`
3. `loadItems()` refreshes with the 18 default items

### Filtering
1. User types in the search box or changes a dropdown
2. `loadItems(filterName, filterPriority, filterMethod)` is called immediately (no debounce)
3. The request includes query parameters for server-side filtering
4. Response replaces the entire `items` state

> **Known issue:** No debouncing on the name search — every keystroke triggers a network request. See [DEVELOPMENT.md](../DEVELOPMENT.md#known-issues).

## API Base Path

The frontend uses relative paths (`/api/items`). In development, Vite's proxy config forwards `/api` to `http://localhost:3001`:

```ts
// vite.config.ts
server: {
  proxy: {
    '/api': 'http://localhost:3001',
  },
}
```
